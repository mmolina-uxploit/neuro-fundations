# Optionals vs Enums para Modelado de Estados

### Análisis de Modelado

| Característica | Optional | Enum (Sum Types) |
| :--- | :--- | :--- |
| **Capacidad** | Binaria (Presencia / Ausencia) | Multiestado (N estados definidos) |
| **Semántica** | Dato faltante o nulo | Estado específico del dominio |
| **Seguridad** | Requiere unwrapping | Forza exhaustividad (switch) |
| **Metadatos** | No permite información extra | Permite valores asociados por estado |

### Observaciones de Diseño

* **Precisión:** El uso de `Optional` para estados complejos genera "estados inválidos representables" (ej. tener datos pero también un error). Los `Enums` hacen que los estados imposibles sean irrepresentables por el sistema de tipos.
* **Manejo de Flujo:** Mientras que el `Optional` solo indica si algo está o no, el `Enum` documenta explícitamente el flujo (ej. `loading`, `success`, `failure`, `empty`).
* **Mantenibilidad:** Los `Enums` facilitan la extensión del código; añadir un nuevo estado obliga al compilador a señalar dónde debe ser gestionado.

### Decisión Arquitectónica

1. **Uso de Enums (Preferencia):**
   * Estados de UI (Carga, Error, Éxito).
   * Resultados de operaciones con múltiples salidas.
   * Flujos de lógica de negocio con reglas específicas.
2. **Uso de Optionals:**
   * Propiedades de objetos que son intrínsecamente opcionales.
   * Parámetros de configuración no obligatorios.
   * Resultados de búsqueda simples donde "no encontrado" es la única alternativa al éxito.
